# 成果物の耐久性と管理方針

開発プロセスで生成される成果物の耐久性と管理方針。「何を残し、何を捨てるか」の判断基準。

関連: `plan-mode.md`、`/rdra` スキル、`/screen-spec` スキル、`/adr` スキル

---

## 成果物の階層

要望→要求→要件→仕様→設計→実装 の階層（IPA『ユーザのための要求定義ガイド 第2版』準拠）。
上流ほどビジネスの「Why」に近く、下流ほど技術の「How」に近い。

| 階層 | 成果物 | 分類 | 耐久性 | 管理方針 |
|------|--------|------|--------|---------|
| 要望→要求→要件 | RDRA 成果物 | Why（事業） | **高** | `docs/rdra/` に永続保存 |
| 設計判断 | ADR | Why（設計） | **高** | `docs/adr/` に永続保存 |
| Contract | Protocol Buffers, OpenAPI 等 | Contract | **高** | コードとして管理 |
| 画面仕様 | 画面仕様書 | What | **中** | `docs/specs/screens/` + Figma リンク |
| 要件→仕様→設計 | 計画ファイル | How | **低** | PR description に統合後削除 |
| 実装 | コード + テスト | Implementation | **最高** | Git 管理。実装後の Source of Truth |

---

## 原則

### 1. Single Source of Truth

コードと仕様書の二重管理を避ける。実装完了後はコードとテストが真実。

古い仕様書は AI Agent にとって特に有害。人間なら「コードが正しい」と判断できるが、AI は渡された情報をすべて等しく信頼する。

### 2. Why は残す、How は捨てる

- **残す**: ADR（設計判断の経緯）、RDRA（事業の構造）
- **捨てる**: 計画ファイル / Design Doc（実装の How）

### 3. Contract はコードに昇華

API 定義は Protocol Buffers / OpenAPI としてコードベース内に存在させる。仕様と実装を繋ぐ恒久的なマッピングドキュメントとして機能する。

### 4. デザインは Figma に委譲

画面仕様書にデザイン詳細（色、フォント、レイアウト）を記載しない。Figma Dev モード URL のみ掲載し、Figma MCP 経由で AI Agent がデザイン情報を取得する。

## 計画ファイルの使い捨て方針

計画ファイルは実装の「思考整理ツール」であり、永続的な仕様書ではない。

| フェーズ | 扱い |
|---------|------|
| 計画中 | `docs/plan/` に保存 |
| 実装開始時 | 計画ファイルをコミット（実装前のスナップショットとして履歴に残す） |
| 実装中 | タスクの受入条件・I/O シグネチャの参照元 |
| PR 作成時 | 計画の要約を PR description の `## Plan` セクションに統合 |
| 実装完了後 | 計画ファイルを削除してコミット（git log で履歴は追える） |

> 「実装完了時点で計画の内容は100%コードに反映されているはず。反映されていなければそれはバグ」

> 計画の「書く過程で要件が整理されること」が本質的価値であり、成果物としてのドキュメント自体は使い捨て。

> 計画ファイルは git log から `git log --diff-filter=D -- docs/plan/` で追跡できる。永続的にツリーに残す必要はない。

## RDRA が必要な場面

| タスク性質 | RDRA | 判断基準 |
|-----------|------|---------|
| 新規システム・大規模新機能 | **必要** | アクター・ゴール・業務フローの構造化が未済 |
| 中規模の新機能 | 状況次第 | 既存 RDRA に追記で済むか、新規分析が必要か |
| 既存機能の改修・バグ修正 | 不要 | コードとテストが仕様。RDRA は参照のみ |

---

## 実装後のドキュメントメンテナンス

画面仕様書など中耐久の成果物は、実装後にコードと重複する情報をコードへのリンクに置き換える。

```markdown
## 表示情報（データ取得）
→ [UserResponse 型定義](../../src/types/user.ts#L10) を参照
```

これにより、型定義の変更時に仕様書との乖離を防ぐ。
