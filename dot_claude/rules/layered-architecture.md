# 三層＋ドメインモデルの設計原則

三層（プレゼンテーション / アプリケーションサービス / データソース）＋ドメインモデルのアーキテクチャにおける設計原則。

---

## 依存の方向

- プレゼンテーション層 → アプリケーションサービス層
- データソース層 → アプリケーションサービス層（の interface 宣言）
- 三層すべて → ドメインモデル

依存性逆転が必要なのはデータソース層の1箇所だけ。Repository の interface をアプリケーションサービス層に置き、データソース層がそれを実装する。

---

## 変換は両端の2箇所に閉じ込める

| 層 | 変換の責務 |
|----|-----------|
| プレゼンテーション層 | 外部形式（JSON 等）↔ ドメインモデルの生成メソッドが求める型 |
| アプリケーションサービス層 | **変換なし**。ドメインモデルの型だけで業務フローを調整 |
| データソース層 | DB 行 / ORM ↔ ドメインモデル |

層ごとに DTO を作って Mapper で変換する構造は過剰。変換コードは外部世界との境界（両端）に閉じ込める。

---

## ユースケース（アプリケーションサービス）の I/O

- **入力**: コマンド — ドメインモデルの生成メソッドの仮引数の型で構成（値オブジェクト＋プリミティブ）
- **出力**: ドメインモデル

コマンドの構造はドメインモデルの生成メソッド側の要求で決まる。リクエストの形に合わせない。

---

## プレゼンテーション層がパースしてよい範囲

単独で完結する値オブジェクト（CustomerId, ProductId 等）まで。

集約の内部要素（OrderItem 等）はプレゼンテーション層で生成せず、集約ルートのファクトリメソッドに委ねる。これにより集約全体の不変条件を1箇所で保証できる。
