# design-note

会話で議論したコード／システムの設計判断・トレードオフ・技術選定の根拠を ideas リポジトリの `design/` ディレクトリに記録する。後から再現するための **設計の Why** が対象。

## Step 0: 対象判定（最重要）

起動前に、本当に design-note の対象かを判定する。**曖昧なまま進めない**。

### 対象に該当する例

- 「この型設計のトレードオフを残して」
- 「PostgreSQL の分離レベル選定の理由をまとめて」
- 「マルチテナント認証の設計方針を記録して」
- 「優先度ありの2値表現を enum vs union で議論した結論を残して」

→ コード／システムの構造・選定理由が中心。`design/` 配下にすでに同種のファイルが並ぶ。

### 対象外の例（このスキルでは扱わない）

| ユーザー発話の例 | 適切な置き場所 |
|------------------|---------------|
| 「このアイデア面白そう、メモして」「思いつきを残して」 | `brainstorming/` に独立ファイル |
| 「AI活用の考え方に追記して」「プロダクト開発論にこの観点足して」 | `job/` 配下の既存「〜の考え方.md」への追記 |
| 「この記事を要約して残して」 | `url-digest` スキル |
| 「日経のトレンドをまとめて」 | `nikkei-trend` スキル |

### 判定フロー

1. 会話内容が **コード／システムの設計選択** か？ → No なら対象外
2. 既存の `job/{...}の考え方.md` に追記する方が文脈的に自然か？ → Yes なら job/ への追記を提案
3. 「アイデア」「着想」「面白そう」のような語彙が出ていないか？ → 出ていれば brainstorming/ を提案
4. ユーザーの指示が曖昧（「メモして」「残して」のみ）なら、**スキル起動前に置き場所を確認する**

判定の結果 design-note の対象でないと分かったら、**スキルを継続せず**、適切な置き場所を提案して終了する。

## Step 1: 前提チェック

まず Bash で `ghq root` を実行してリポジトリルートを取得する。以降のステップではその出力を `{GHQ_ROOT}` として使用する。

次に ideas リポジトリの存在を確認する。

```bash
[ -d {GHQ_ROOT}/github.com/RyoMa99/ideas ] && echo "OK" || echo "NG: ideas リポジトリが見つかりません"
```

ideas リポジトリが存在しない場合は動作しない。ユーザーに `ghq get github.com/RyoMa99/ideas` を案内して終了する。

## Step 2: ファイル名の決定

会話のトピックを日本語で簡潔に表すファイル名を決める。

- 形式: `{トピックの説明}.md`（スペースなし）
- 例: `優先度のある2値設計パターン.md`、`マルチテナント認証設計.md`
- 既存ファイルと重複する場合はユーザーに確認する

## Step 3: まとめの作成

会話から Markdown を作成する。**簡潔さより再現性を優先する**（後から読んだ人が議論の経緯と結論を理解できること）。

### 構成テンプレート

```markdown
# {トピック}

> {1〜2行の文脈説明。なぜこの設計判断が必要になったか}

---

## {セクション}

（比較表・コード例・フロー図を積極的に使う）

## まとめ / 判断フロー

決定の要点を箇条書きやフローで整理する。
```

セクション構成は内容に応じて柔軟に変える。今回の会話で登場したコード例は積極的に再掲する。

## Step 4: 保存

`{GHQ_ROOT}/github.com/RyoMa99/ideas/design/` に保存する。ディレクトリがなければ `mkdir -p` で作成する。

Write ツールでファイルを書き出し、完了後に保存したフルパスをユーザーに伝える。
