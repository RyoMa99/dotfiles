# ADR（Architecture Decision Record）

ソフトウェアやシステムの設計における意思決定を構造化して記録する。
あとから読む人が「なぜこうなったのか」を理解できることが最優先。

関連: `artifacts.md`（ADR は高耐久の成果物）

---

## ADR を書くべきタイミング

以下の条件に1つでも該当すれば ADR を作成する。

```
この判断は ADR に残すべきか？
    │
    ├─ 可逆性が低い（変更コストが高い）→ Yes
    │   例: DB 選定、認証方式、API 設計、アーキテクチャパターン
    │
    ├─ 複数の選択肢を比較して決定した → Yes
    │   例: plan-mode Phase 2 の設計選択肢比較、RDRA の to-be 選択
    │
    └─ 可逆で、比較検討も不要だった → No
        例: ライブラリの軽微な使い方、命名の選択
```

### ワークフロー上の発火ポイント

ADR は以下の場面で自然に発生する。各スキルが ADR 作成を提案する。

| 発火ポイント | 条件 | 呼び出し元 |
|-------------|------|-----------|
| **RDRA Step 3** | to-be 候補を複数比較して選択した | `/rdra` |
| **plan-mode Phase 2** | 設計選択肢を比較して決定した | `plan-mode.md` |
| **implementation 中** | 計画外の設計判断が発生した | `/implementation` |
| **session-retrospective** | 振り返りで重要な設計判断を検出 | `/session-retrospective` |

> 計画ファイルは使い捨て（`artifacts.md` 参照）。計画に書かれた設計判断の Why は、計画削除時に消える。重要な判断は ADR に抜き出すこと。

### スコープ

ADR が対象とする判断のスコープは広い。「Architecture」に限定しない。

| レベル | 例 |
|--------|-----|
| **要求レベル** | アクター分割の理由、to-be フロー選択の根拠、要件の優先順位判断 |
| **設計レベル** | DB 選定、認証方式、API 設計、アーキテクチャパターン |
| **技術レベル** | フレームワーク選定、ライブラリ選定、デプロイ戦略 |

> RDRA 分析中の重要な判断（なぜこのアクター分割か、なぜこの to-be フローか）も ADR で記録する。

## Step 1: 会話コンテキストの判定

現在の会話に以下の情報が十分に含まれているか判定する。

- **課題**: 何が問題で、なぜ意思決定が必要だったか
- **選択肢**: どんな代替案を検討したか
- **却下理由（Why not）**: 採用しなかった選択肢の却下の決定打
- **決定事項**: 何に決めたか
- **トレードオフ**: 何を得て、何を捨てたか
- **影響**: この決定の結果として何が必要になるか（後続タスク）

### 十分な場合

会話の内容をもとに ADR のドラフトを生成し、ユーザーに確認する。
過不足があればユーザーのフィードバックを受けて修正する。

### 不十分な場合

対話的に情報を集める。以下の順序で質問する（既に会話にある情報はスキップ）。

1. **課題**: 「どんな問題・状況があって、この意思決定が必要になりましたか？」
2. **選択肢と比較**: 「検討した選択肢を教えてください。それぞれのメリット・デメリットは？」
3. **決定事項**: 「最終的に何に決めましたか？」
4. **トレードオフ**: 「この決定で得られるものと、捨てるもの（妥協すること）は？」
5. **影響**: 「この決定の結果として、追加で必要になる作業や対応はありますか？」

一度に全部聞かず、会話の流れに合わせて自然に引き出す。

## Step 1.5: 過去 ADR との関連チェック

新しい ADR を書く前に、`docs/adr/` に既存の ADR があるか確認する。

```bash
ls docs/adr/*.md 2>/dev/null
```

### 既存 ADR がある場合

今回の決定が**過去の ADR の決定事項を覆す・変更する**ものか判定する。

```
今回の決定は過去の ADR と関連するか？
    │
    ├─ 過去の決定を覆す / 方針変更
    │   → 該当 ADR の Status を「Deprecated by [ADR-xxx]」に更新
    │   → 新しい ADR の背景に「ADR-xxx で決定した〇〇を見直す」と明記
    │
    ├─ 過去の決定を補完・拡張
    │   → 過去の ADR はそのまま（Status 変更なし）
    │   → 新しい ADR の背景に「ADR-xxx を踏まえ〇〇を追加決定」と明記
    │
    └─ 関連する過去 ADR なし
        → そのまま Step 2 へ
```

### Status 更新の実行

Deprecated にする場合、**過去 ADR の Status 行のみ**を Edit ツールで更新する。本文は改変しない。

```markdown
# 更新前
- **Status**: Accepted

# 更新後
- **Status**: Deprecated by [ADR-012]
```

## Step 2: ADR の生成

以下のテンプレートに沿って ADR を生成する。

```markdown
# {タイトル}

- **Date**: YYYY-MM-DD
- **Status**: Proposed / Accepted / Deprecated by [ADR-xxx]

# Summary

## 課題
<!-- この決定や変更をしなくてはいけない状況、問題の定義を簡潔に -->

## 決定事項
<!-- 何をすることに決定したのかを簡潔に -->

# Detail

## 背景
<!-- この決定や変更をしなくてはいけない状況、問題の定義を詳細に -->

## 検討した選択肢
<!-- 各選択肢を個別に列挙し、比較表で整理する -->
<!-- | 選択肢 | メリット | デメリット | 所感 | -->

## 却下した選択肢と理由（Why not）
<!-- 採用しなかった各選択肢について、却下の決定打を1-2文で明記する -->
<!-- Agent が将来「改善」として却下済みの設計に回帰するのを防ぐ -->

## 議論・考察
<!-- どのような検討の流れで結論にたどり着いたのかを詳細に -->
<!-- コード例、構成図、コスト概算など、議論の材料も含める -->

# トレードオフ

## 得られるもの

## 捨てるもの

# 影響
<!-- この決定の結果として必要になる後続タスク・対応事項 -->
<!-- 例: メール送信基盤の構築が必要、認証トークンテーブルの追加 -->

# 参考
<!-- 外部資料、RFC、OWASP、技術ブログ等の根拠があれば記載 -->
```

### 書き方のポイント

- **Summary は独立して読めるようにする** — Detail を読まなくても決定の概要がわかること
- **Detail は再現性を優先** — 簡潔さより「なぜその結論に至ったか」の経緯が追えること
- **検討した選択肢は独立セクション** — 議論・考察に埋もれないよう、選択肢を先に列挙してから考察に入る
- **影響は後続タスクを明記** — トレードオフ（何を捨てたか）とは別に、「この決定の結果として何が必要になるか」を記録する
- **Why not は決定打を明記** — 「デメリットがある」では不十分。「○○の理由で致命的だった」のように、却下の決定打を書く。Agent が将来「改善」として却下済みの設計に回帰するのを防ぐ
- **Status でライフサイクル管理** — 状況が変わったら既存 ADR の Status を `Deprecated by [ADR-xxx]` に更新し、新しい ADR を書く。既存 ADR の本文は改変しない
- **セクションの省略** — 参考資料がなければ「参考」セクションは省略してよい。影響が自明なら「影響」も省略可

---

## Step 3: 出力

生成した ADR を会話上に表示する。

ユーザーから「ファイルに書き出して」「保存して」等の指示があった場合のみ、指定された場所にファイルとして保存する。指定がなければプロジェクトの `docs/adr/` を提案する。
