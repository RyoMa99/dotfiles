# 画面仕様書（Screen Spec）

画面単位・コンポーネント単位の仕様書を Markdown 形式で作成する。

関連: `artifacts.md`（成果物の耐久性 — 画面仕様書は「中」）、`/rdra`（上流の要件定義）

---

## When to Use

- `/screen-spec` コマンドを実行
- 新規画面の設計時
- 既存画面の仕様整理時
- フロントエンド・バックエンド間の認識合わせ

## 管理方針

- **配置先**: プロジェクトの `docs/specs/screens/` に保存
- **耐久性**: 中。デザイン詳細は Figma に委譲し、仕様書は構造・振る舞いに集中することで鮮度を保つ
- **コードとの重複回避**: 実装後、コードと重複する情報はコードへのリンクに置き換える

---

## Step 1: 画面の特定

対象画面を特定し、以下を確認する。

- URL パス
- 関連する RDRA 要件（REQ-xxx）やビジネスユースケース（BUC-xxx）があれば紐づけ
- Figma デザインの有無

---

## Step 2: 仕様書の作成

以下のテンプレートに沿って作成する。

```markdown
# 仕様: [画面名]

## URL Path
/path/to/page

## 関連要件
- REQ-xxx: [EARS 要件の概要]
- BUC-xxx: [ビジネスユースケースの概要]

## Figma / デザイン
- [Figma Dev モード URL]（ある場合）

## 表示情報（データ取得）

### [コンポーネント / セクション名]

| 項目 | 型 | 説明 | 例 |
|------|-----|------|-----|
| ... | ... | ... | ... |

## アクション・インタラクション

### バリデーション

| 項目 | 入力種別 | 必須 | バリデーション | エラーメッセージ |
|------|---------|------|--------------|----------------|
| ... | ... | ... | ... | ... |

### [アクション名]

[ボタン / リンク] クリック時の処理:
- 処理内容
- 成功時の遷移先
- 失敗時の振る舞い

### エラーハンドリング

| エラー | ステータスコード | 事象 | 振る舞い |
|--------|----------------|------|---------|
| ... | ... | ... | ... |

## アナリティクス（任意）

| イベント | event_name | パラメータ |
|---------|-----------|-----------|
| ... | ... | ... |
```

### 書き方のポイント

- **デザインは Figma に委譲**: 色、フォント、レイアウトの詳細は記載しない。Figma Dev モード URL のみ
- **振る舞いに集中**: バリデーション、遷移、エラーハンドリングなど、実装に必要な情報を記載
- **データの型と例を明記**: AI Agent がコード生成する際の精度向上に直結
- **エラーメッセージを網羅**: ユーザー向けメッセージは仕様として明示

---

## Step 3: レビューと保存

作成した仕様書を `AskUserQuestion` でユーザーにレビューを依頼する。

承認後、`docs/specs/screens/[画面名].md` に保存する。ファイル名はケバブケースの英語（例: `user-registration.md`）。

---

## 実装後のメンテナンス

実装完了後、コードと重複する情報はコードへのリンクに置き換える。

```markdown
## 表示情報（データ取得）
→ [UserResponse 型定義](../../src/types/user.ts#L10) を参照
```

バリデーションルールやエラーメッセージなど、コードに明示的に表現されにくい情報は仕様書に残す。
