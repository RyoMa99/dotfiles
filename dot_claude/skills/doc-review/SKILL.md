# Doc Review Skill

スライドや発表原稿を「緊張と弛緩モデル」に基づいてレビューする。

## References

- [note添削用のGemを作りました](https://note.com/gimupop/n/nd14f5467170d) by gimupop（柳川慶太氏）

## When to Use This Skill

Trigger when user:
- `/doc-review` コマンドを実行
- 「スライドをレビューして」「発表原稿を添削して」と依頼
- 「プレゼンのフィードバックがほしい」と依頼

## 緊張と弛緩モデル

面白いプレゼンに共通するリズム：

```
┌─────────────────────────────────────────────────────────┐
│  緊張（Tension）                                        │
│  ─────────────────                                      │
│  聴衆を「えっ？」と思わせ、関心を引くフェーズ           │
│                                                         │
│  例:                                                    │
│  - 意外な事実、逆説的な主張                             │
│  - 問題提起、課題の深刻さ                               │
│  - 「実は〜」「よくある誤解は〜」                       │
│                                                         │
│  ⚠️ これがないと → 退屈、聞く理由がない                │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  弛緩（Release）                                        │
│  ─────────────────                                      │
│  聴衆を救い、満足させるフェーズ                         │
│                                                         │
│  例:                                                    │
│  - 解決策、具体的なアクション                           │
│  - ポジティブなメッセージ、希望                         │
│  - 「でも、こうすれば〜」「結果として〜」               │
│                                                         │
│  ⚠️ これがないと → 不快感、モヤモヤが残る              │
└─────────────────────────────────────────────────────────┘
```

## レビュー観点（5項目）

### 1. Spice（緊張）⚡

聴衆の関心を引くフックがあるか。

**チェックポイント:**
- [ ] 冒頭で「えっ？」と思わせる要素があるか
- [ ] 問題提起が明確か
- [ ] 意外性、逆説、鋭い指摘があるか
- [ ] 「なぜこの話を聞くべきか」が伝わるか

**改善例:**
```
❌ 「今日はXについて話します」
✅ 「Xを導入した企業の70%が失敗しています。なぜでしょう？」
```

### 2. Release（弛緩）🎯

聴衆を救う解決策・結論があるか。

**チェックポイント:**
- [ ] 問題に対する解決策が提示されているか
- [ ] 聴衆が「聞いてよかった」と思える結論か
- [ ] 具体的なアクションが示されているか
- [ ] ポジティブな締めくくりになっているか

**改善例:**
```
❌ 「以上が問題点でした」
✅ 「この3つを実践すれば、問題の80%は解決できます」
```

### 3. Human Context（人間味）👤

AIには書けない、あなただけの視点があるか。

**チェックポイント:**
- [ ] 個人的な経験、失敗談、学びが含まれているか
- [ ] 抽象概念を身体感覚のある言葉で説明しているか
- [ ] 「なぜあなたがこれを話すのか」が伝わるか
- [ ] 聴衆が共感できるストーリーがあるか

**改善例:**
```
❌ 「ベストプラクティスによると〜」
✅ 「私も最初は失敗しました。3ヶ月間、毎日エラーと格闘して気づいたのは〜」
```

### 4. 構成・フロー📐

緊張と弛緩のリズムが適切か。

**チェックポイント:**
- [ ] 緊張→弛緩の波が複数回あるか（単調でない）
- [ ] 各セクションに小さな緊張と弛緩があるか
- [ ] 最後に向けて盛り上がる構成か
- [ ] 聴衆が迷子にならない流れか

**理想的な構成:**
```
1. 導入    [緊張] 問題提起、意外な事実
2. 背景    [弛緩] 共感、現状説明
3. 本題    [緊張] 深掘り、核心に迫る
4. 解決策  [弛緩] 具体的な方法
5. 結論    [緊張→弛緩] まとめ、行動喚起
```

### 5. リスク管理🛡️

炎上・誤解のリスクがないか。

**チェックポイント:**
- [ ] 特定の個人・組織を不必要に批判していないか
- [ ] 機密情報、社外秘が含まれていないか
- [ ] 誤解を招く表現がないか
- [ ] 根拠のない断定をしていないか

## 実行フロー

### Step 1: 資料の確認

ユーザーからスライドまたは発表原稿を受け取る。

対応形式:
- マークダウン
- テキスト
- 画像（スライドのスクリーンショット）
- PDF

### Step 2: 5項目でレビュー

各観点でスコアリング（1-5）と具体的なフィードバック。

### Step 3: レビュー結果を出力

```markdown
## レビュー結果

### スコア

| 観点 | スコア | コメント |
|------|--------|----------|
| Spice（緊張） | 3/5 | 冒頭のフックが弱い |
| Release（弛緩） | 4/5 | 解決策は明確 |
| Human Context | 2/5 | 個人的な視点が少ない |
| 構成・フロー | 4/5 | 流れは良い |
| リスク管理 | 5/5 | 問題なし |

**総合: 18/25**

### 良い点 ✅
- [具体的な良い点]

### 改善提案 💡

#### 1. [改善点のタイトル]
**現状:** [現在の記述]
**提案:** [改善案]
**理由:** [なぜ改善すべきか]

#### 2. ...

### 全体コメント
[総評]
```

### Step 4: 改善案の提示

具体的な書き換え案を提示（文体は変えず、構成・展開を磨く）。

## 使用例

```
/doc-review

[スライドの内容をペースト]
```

または

```
このスライドをレビューして

# タイトル
...
```

## 重要なルール

1. **文体は変えない** - 個性を尊重し、構成・展開のみ改善
2. **具体的に指摘** - 「もっと面白く」ではなく「ここに緊張を追加」
3. **良い点も伝える** - 改善点だけでなく、良い点も明示
4. **押し付けない** - 提案であり、最終判断は著者に委ねる
