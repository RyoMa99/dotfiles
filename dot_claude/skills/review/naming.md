# Naming Review

kawasima氏の「命名のプロセス」に基づく進化的アプローチでコードの命名をレビューするスキル。

関連: @~/.claude/rules/robust-code.md（型による予防的設計、Value Object）

> 良い命名は単一ステップではなく**プロセス**である。
> 最初から完璧を目指すのではなく、段階的に改善していく。

命名は設計であり、コードの読みやすさに直結する。プログラマはプログラミング全体の60-70%をコードを読むことに費やしている。

---

## スコープ規律

**ユーザーが明示的に提示した範囲だけをレビューする**。ファイル・ディレクトリ・貼り付けられたコードスニペットの和集合が対象。

- ユーザーが「リポジトリ全体」「横断的に」など広範囲を明示していない限り、周辺コードへの再帰スキャンは行わない
- スコープ外の識別子が目に入っても言及しない（集中を保つ）
- スコープが曖昧な場合は `AskUserQuestion` で確認してからレビューを開始

---

## 7段階 × 3フェーズ構造

7段階の命名進化は、3つのフェーズにグループ化される。**フェーズ境界には権限ゲートがある**（Phase 2/3 への提案はユーザーの明示的な許可を要する）。

| Phase | 段階 | 性質 | 権限 |
|-------|------|------|------|
| **Phase 1: Insight → Name** | Missing → Nonsense → Honest → Honest and Complete | 純粋な命名変更。構造変更なし | **不要**（連続的に提案してよい） |
| **Phase 2: Name → Structure** | Honest and Complete → Does the Right Thing | メソッド/クラス分割などの構造的リファクタリングが必要 | **要**（「構造変更も含めて見るか?」を確認） |
| **Phase 3: Combine for Design** | Does the Right Thing → Intent → Domain Abstraction | 呼び出し元の読解・ドメイン文脈の理解が必要 | **要**（「ドメイン設計まで踏み込むか?」を確認） |

> フェーズ境界で立ち止まる理由: Phase 2 は動作に影響する構造変更、Phase 3 はコードベース全体の文脈を要するため、ユーザーの投資判断が必要。

### Stage 1-2: Missing → Nonsense

名前のない概念を見つけ、まずは抽出する。

**長いものに注目**: 長いメソッド・クラス・ファイル、長い引数リスト、複雑な式
**塊を探す**: ひとまとまりの文、明確でない式、よく一緒に渡されるパラメータ

リファクタリング手法:
- **メソッド抽出** - 文の塊に名前を付ける
- **変数導入** - 式に名前を付ける
- **パラメータオブジェクト導入** - パラメータ集合に名前を付ける

**applesauce マーカー**: 誤解を招く既存名 (`-Manager`, `-Handler`, `process()` 等) に対しては、改善提案の中間形として `applesauce` のような明らかに無意味な名前を経由させることを推奨する。「部分的にマシに見える」ことが思考停止を生むため、誤った安心感をいったん剥がす手続きが有効。`-Manager` が何かを言っているように見えても実質は何も言っていない — `process()` と同じくらい Nonsense 候補として扱う。

**例外**: 意味が1行の文脈から明白な1文字変数 (`d`, `r` 等) は applesauce を経由せず直接 Honest に進めてよい。applesauce は「誤解を招く名前」と「抽出された塊」に適用する手続きである。

### Stage 3: Nonsense → Honest

メソッドの中心となる1つの概念を理解し、実直な名前にする。コード内のパターン（システムコンポーネント変数、繰り返されるパターン、戻り値の使われ方）を探す。

**重要**: **何であるか**ではなく**何をするか**で命名する。

**不確実性マーカー**: 作用の全体像が把握できていない段階では、以下のマーカーで不確実性を**名前自体に表明する**ことを推奨する。

- `probably_` プレフィックス: 推測に基づく部分（テストで確認後に除去）
- `_AndStuff` サフィックス: 未把握の残余作用（追跡が進んだら具体化）

```typescript
// 全体像が不明な段階での正直な Honest 名
probably_doSomethingEvilToTheDatabase_AndStuff()
```

専門的に見える誤解的な名前より、正直な不確実性表明の方が優れている。

### Stage 4: Honest → Honest and Complete

メソッドがしている**すべて**を名前で表現する。名前が長くなるのは**設計の問題を示すシグナル**。次のステージで責務を分割する。

**反復手順**:
- **既知を拡張**: 名前に含まれていない作用を1つずつ追加
- **未知を狭める**: `_AndStuff` がカバーしない範囲を特定し、サフィックスを具体化
- **ゴール**: `probably_` と `_AndStuff` が両方外れる状態。名前だけ見れば挙動が復元できるレベル

```
probably_doSomethingEvilToTheDatabase_AndStuff()
→ parseXmlAndStoreFlightToDatabaseAndLocalCacheAndBeginBackgroundProcessing()
```

この段階では名前の長さは美徳。短縮は Intent 段階で行う。

### Stage 5: Does the Right Thing

責務を分割し、単一責務の原則に従う。

1. 名前から排除したい部分を特定
2. その部分が他と無関係か確認
3. 構造的リファクタリングを実施

### Stage 6: Intent（意図）

**何をしているか**から**なぜ必要か**へ転換する。

見るべき対象:
- **メソッド**: 事後条件、実現される変換、ビジネスプロセス
- **クラス**: 共通責務、現実世界の対応物
- **変数**: 同じ型の他インスタンスとの違い

避けるべきパターン: 初期条件で命名、コンピュータ科学用語のみ使用、ドメイン文脈を欠いた命名。

### Stage 7: Domain Abstraction

共有コンテキストと統一されたドメイン概念を形成する。

探すべきパターン:
- **クラス群**: 類似の名前、Manager/Transform/~er で終わる名前、Feature Envy
- **メソッド群**: 一緒に渡される複数パラメータ、同じデータへの操作、類似の名前
- **フィールド/パラメータ**: 型名と同じ変数名、許容値や解釈が必要な名前

Value Object の導入で、プリミティブ型への執着を解消する。

## 深度シグナル（ユーザー発話 → 停止位置）

ユーザー発話から改善の深さを判定し、レビュー範囲を過不足なく合わせる。明示的な深度シグナルがあるなら、そのレベルを超えた提案は「候補として列挙のみ」に留める。

| ユーザー発話 | 停止位置 | 提案範囲 |
|------------|---------|---------|
| "急いでる" / "bug fix" / "とりあえず" | Honest（Phase 1 中段） | 誤解を招く名前の是正のみ |
| "改善して" / "rename" / 文脈なし | Honest and Complete（Phase 1 終端） | Phase 1 全体を連続的に提案 |
| "リファクタリング" / "構造も見て" | Does the Right Thing（Phase 2） | Phase 1 + Phase 2 の構造分割案 |
| "設計から見直したい" / "ドメイン的に整理" | Intent / Domain Abstraction（Phase 3） | 全フェーズ。Value Object 抽出も提案 |

> 「ストーリーの現状に必要な最小ステップで止める」— 後続の担当者のために名前を未完成のまま残すのは欠陥ではなく設計。

## チェックリスト

- [ ] 名前のない概念が放置されていないか
- [ ] 誤解を招く名前はないか
- [ ] メソッド名は主要な作用を反映しているか
- [ ] 名前が長すぎる（責務が多すぎる）サインはないか
- [ ] 何をするかではなく、なぜ必要かを表現しているか
- [ ] ドメイン用語を適切に使用しているか
- [ ] プリミティブ型への執着はないか（Value Object化の検討）

## Anti-Patterns（やってはいけない事）

| 禁止事項 | 理由 |
|----------|------|
| `process()` をいきなり `importFlightData()` に置換提案 | Missing から Intent への飛躍。Honest and Complete を経由すべき |
| 1回の提案で5つ以上のリネームを束ねる | 各リネームは独立した洞察。段階的に提案すべき |
| クラス本体を読まずにクラス名を提案 | 読んでいない内容に対して Honest にはなれない |
| `probably_` / `_AndStuff` を「プロ感がない」としてスキップ | 誤解を招く"プロっぽい"名前がバグを生む。正直な不確実性の方が健全 |
| Honest and Complete の段階で短い名前を提案 | 長い名前が正解のフェーズ。短縮は Intent 段階 |
| `-Manager` / `-Handler` / `process()` が「一応何か言っている」として applesauce をスキップ | 何も言っていない。誤解的であることを受け入れる手続きを省かない |
| "いつ呼ばれるか" で命名 (`onInit`, `preLoad`) | 「何をするか」または「なぜ必要か」で命名する |
| 型名と同じ変数名 (`GridSquare gridSquare`) | このインスタンスを他と区別する情報で命名する |
| Intent 段階で CS 用語 (`Transformer`, `Processor`, `Handler`) を使う | 事業側が理解するドメイン用語を使う |
| スコープ外の識別子まで「ついでに」指摘 | レビューはユーザー提示範囲に限定。範囲外への脱線は集中を削ぐ |
| 「完成形の名前」だけ提示して中間形を示さない | 名前はプロセスで育つ。中間段階の名前も提案に含める |

## 実行フロー

### 1. 対象と深度の特定

```
対象: $ARGUMENTS
```

- 引数がない場合はユーザーに確認（スコープ規律に従う）
- ユーザー発話から**深度シグナル**を読み取り、停止位置を決定（「深度シグナル」セクション参照）
- 深度が不明な場合は `AskUserQuestion` で「Phase 1 のみか、構造・ドメインまで見るか」を確認

### 2. コード読み取り

対象ファイル/ディレクトリの公開API（関数、クラス、メソッド）を特定。スコープ外への再帰は行わない。

### 3. 各識別子を7段階で評価

| 段階 | 名前 | 問題の兆候 |
|------|------|-----------|
| 1 | Missing | 抽出すべき概念が埋もれている |
| 2 | Nonsense | temp, data, info 等の汎用名 |
| 3 | Honest | 副作用が名前に反映されていない |
| 4 | Honest and Complete | 名前が長い（責務過多のサイン） |
| 5 | Does the Right Thing | 単一責務 |
| 6 | Intent | 目的を表現 |
| 7 | Domain Abstraction | ドメイン用語として統一 |

### 4. 問題パターンの検出

- 誤解を招く名前
- プリミティブ型への執着（Value Object化の候補）
- Manager/Handler/Processor 等の曖昧なサフィックス
- Feature Envy（他オブジェクトへの過度なアクセス）

### 5. フェーズ別に改善提案を出力

深度シグナルで決めた停止位置までのフェーズだけを詳細提案として出し、それより深いフェーズは「候補として列挙のみ」に留める。

**Phase 1（権限不要）**: 純粋な命名提案として詳細に展開
- 各識別子の現在段階
- 問題点
- 中間形を含む改善案（`applesauce` → `probably_...` → Honest → Honest and Complete の流れを示す）
- 推奨案と理由

**Phase 2（構造変更を要する）**: 権限ゲート
- 深度シグナルが「リファクタリング」以上のときのみ詳細案を出す
- それ未満のときは「構造分割の候補」として1行ずつ列挙に留め、展開は次回のレビューに委ねる

**Phase 3（ドメイン文脈を要する）**: 権限ゲート
- 深度シグナルが「設計から見直し」のときのみ詳細案
- 未達のときは「Value Object 候補」「ドメイン語彙の統一候補」として表形式で列挙のみ

### 6. サマリー出力

- 評価した識別子数と問題数
- フェーズ別分布（Phase 1 / 2 / 3 のそれぞれの候補数）
- 本レビューで採用した深度シグナルと停止位置
- 次回深度を上げる場合に期待できる追加の改善候補（ユーザーの意思決定材料）
