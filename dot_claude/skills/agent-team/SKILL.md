# Agent Team 編成パターン

Claude Code の Agent Team を組む際の編成パターンと選択指針。

---

## いつ Agent Team を使うか

- 不可逆な設計判断（アーキテクチャ選定、技術スタック変更）
- 複数の視点からの検証が価値を持つ議題
- 単一エージェントの自己レビューではバイアスが懸念される場面

コストが高いため、日常的な実装タスクには使わない。

---

## 編成パターン

### 1. 役割ベース（Role-based）

技術領域で分担する。実装タスクの並列化に向く。

| メンバー | 責務 |
|---------|------|
| Frontend | UI・UX・コンポーネント設計 |
| Backend | API・ビジネスロジック・データモデル |
| QA / Reviewer | テスト戦略・品質検証 |

**向く場面**: 実装フェーズ、明確に分割可能なタスク
**弱み**: 各メンバーが自分の領域に閉じ、横断的な洞察が出にくい

### 2. 思考スタイルベース（Thinking-style-based）

異なる思考の「生産的な摩擦」で多角的な検討を行う。

| メンバー | 問い |
|---------|------|
| Pragmatist（実用主義） | 最もシンプルなアプローチは？ |
| Skeptic（懐疑主義） | 何がうまくいかない可能性がある？ |
| Idealist（理想主義） | 6ヶ月後、この設計を誇れるか？ |
| Connector（接続者） | これは既存の○○と似ていないか？ |

**向く場面**: アーキテクチャ議論、不可逆な技術選択
**弱み**: コストが高い、実装タスクには過剰

### 3. ステークホルダーベース（Stakeholder-based）

異なる利害関係者の視点でプロダクト判断を検討する。

| メンバー | 視点 |
|---------|------|
| Engineer | 技術的実現可能性・保守性 |
| User | 使いやすさ・課題解決 |
| Business | 収益性・市場性・スケーラビリティ |

**向く場面**: 機能の優先度判断、プロダクト方針の検討
**弱み**: 技術的な深掘りには不向き

---

## パターン選択の判断基準

| 場面 | パターン | 発動 |
|------|---------|------|
| 実装フェーズ | 役割ベース | 自動（タスク分割時） |
| 機能設計（Plan モード） | ステークホルダーベース | 自動（Plan モード時） |
| 設計の深掘り | 思考スタイルベース | ユーザーからの明示的な指示 |

---

## 運用上の注意

- 各メンバーには極端な立場を徹底させ、ファシリテーター（リーダー）が合成する
- 結論を出すのはチームではなくファシリテーター。チームは視点を提供する
- 議論の結果は `docs/` 等に ADR（Architecture Decision Record）として記録する
