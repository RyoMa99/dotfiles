# Log Design Skill

アプリケーションログの設計指針を提供するスキル。
新規のログ設計、既存ログのレビュー、オブザーバビリティ戦略の策定に使う。

## 実行フロー

1. 対象コードまたは設計方針のヒアリング
2. 階層モデル（L0-L5）でログの目的を上位の非機能要求まで遡って確認
3. 5原則に基づく設計レビューまたは提案
4. チェックリストによる最終確認

---

## ログ設計の階層モデル（L0-L5）

ログ設計はビジネス目標から具体ツール選定まで階層化して整理する（kawasima 氏の階層モデルに準拠）。

| 層 | 名称 | ログ文脈での例 |
|----|------|---------------|
| L0 | ビジネス目標 | 24時間365日のサービス提供／規制準拠による事業継続 |
| L1 | 非機能要求 | 年間稼働率 99.99%／インシデント RTO 30分／SOC2 準拠 |
| L2 | 品質特性 | 可用性・保守性・セキュリティ・運用性・監査性 |
| L3 | 戦術 | 異常を検知する／問題特定を支援する／攻撃から回復する／証跡を保全する |
| L4 | 技法 | ロギング（Canonical Log Line / 構造化ログ / MDC）／メトリクス／トレース |
| L5 | ツール | OpenTelemetry / Loki / Datadog / CloudWatch Logs |

### セミラティス構造（多対多）

ログ（L4 技法）は単一の戦術には属さない。同じロギング技法が複数の戦術を同時に支える。

- ロギング（L4） → 「異常を検知する」「攻撃から回復する」「問題特定を支援する」「証跡を保全する」（L3 複数）
- 監査ログ（戦術カテゴリ） → 可用性・セキュリティ・コンプライアンス（L2 複数の品質特性に貢献）

**含意**: ログ1本を設計するときに「どの非機能要求のために、どの戦術として出すのか」を必ず複数の観点で問う。単一目的だけ満たして他を見落とすと、後で同じ事象を別目的で再ログ化する破目になり粒度・命名が割れる。

### トレーサビリティ

ログ設計のレビュー時には、各ログ出力から **L1 非機能要求まで遡れるか** を確認する。遡れないログは「なぜ存在するか」が言語化されていないため、削除候補かカテゴリ未分類のいずれか。

参考: [アーキテクチャ設計の階層モデル - kawasima](https://scrapbox.io/kawasima/アーキテクチャ設計の階層モデル)

## 5つの設計原則

### 原則 1: 何をログに残すか決め、継続的に見直す

ログの不足と過剰は両方コストになる。不足はインシデント調査を妨げ、過剰はパフォーマンス劣化とストレージコスト増を招く。

**Canonical Log Line パターン**を推奨する。**「実行単位」**（HTTP リクエスト / バッチジョブ / キュー消費 / cron 起動 / gRPC 呼び出しなど、開始と終了が定義できる処理スコープ）の最後に、その単位に関する全コンテキストを1行の構造化ログに集約して出力する（Stripe・Amazon の手法）。

- ミドルウェア / インターセプター / テンプレートメソッドでコンテキストを蓄積し、処理完了時にまとめて出力する
- 処理の各ステップで個別にログを出すのではなく、**1 実行単位 = 1 集約エントリ**を基本にする
- ソースコード上のどこでログを出すかを明確にし、重複出力を防ぐ

**実行単位ごとの集約位置**:

| 実行単位 | 集約タイミング | 集約位置 |
|---------|--------------|---------|
| HTTP リクエスト | レスポンス送信直前 | サーバーミドルウェア |
| gRPC 呼び出し | 応答返却直前 | ServerInterceptor |
| バッチジョブ | ジョブ完了時 | テンプレートメソッド末尾 |
| キュー消費 | メッセージ ack 直前 | コンシューマラッパー |
| cron 起動 | スクリプト終了時 | finally 句 / atexit ハンドラ |

**Canonical と他カテゴリの重複出力**: 戦術カテゴリの一部は Canonical に吸収できる場合と、独立して出すべき場合がある。

| 状況 | 推奨 |
|------|-----|
| 業務イベントが「実行単位の最終結果」と一致する（例: HTTP の `order_placed`、バッチの `import_completed`） | Canonical に統合（`event` フィールドで表現） |
| 業務イベントが実行単位中に複数回発生する（1 リクエストで複数注文、バッチの各チャンク KPI） | 独立して都度出力（KPI の粒度維持のため） |
| 障害追跡（行単位エラー、リトライ詳細） | 都度独立。Canonical には件数とサマリのみ集約 |
| リモート呼び出し（外部 API 単発） | 都度独立。Canonical には呼び出し件数 / 累積レイテンシを集約 |
| 外部 API 呼び出しが業務イベントの中核を成す（例: 決済 API が注文確定の核 / ML 推論 API が予測機能の核 / 外部検索 API が検索機能の核 / メッセージング基盤が通知機能の核） | **両建て**。外部呼び出しは個別追跡のため都度独立（責任分界・突合の証跡）、業務イベントは Canonical に統合（同一実行単位で2系統出る前提） |

**ログ用途の分類**: ログは目的ごとに「戦術カテゴリ」に分けると過不足の判断がしやすい。Canonical Log Line は実行単位を貫く集約パターンとして、複数カテゴリを束ねる。各カテゴリの位置づけは § ログの戦術分類（10カテゴリ）を参照。

**判断基準**: 「このログがなかったら、インシデント調査で何が困るか？」と問いかける。答えられないログは削除候補。

### 原則 2: 構造化ログを使う

メッセージ文字列にデータを埋め込むのではなく、フィールドとして分離する。

```
# Bad: メッセージにデータを埋め込む
logger.info(f"User {user_id} placed order {order_id} for ${amount}")

# Good: フィールドとして構造化
logger.info("order_placed", user_id=user_id, order_id=order_id, amount=amount)
```

構造化ログの利点:
- ログ管理システムでのフィールド単位のフィルタリング・インデックス作成
- メッセージ文字列の変更がフィルタやアラートを壊さない
- 集計・メトリクス変換が容易

フィールド名は定数や enum で管理し、タイポや不整合を防ぐ。

**MDC（Mapped Diagnostic Context）パターン**: リクエスト ID・ユーザー ID・テナント ID のような全ログに付けたい横断的コンテキストは、関数引数で引き回すのではなく実行コンテキストに格納して自動付与する。ハンドラ／ミドルウェアの入口でセットすれば、配下のすべてのログ出力で透過的に参照される。詳細は § 実装パターン を参照。

### 原則 3: イベントを分類する

**ログレベルの使い分け**:

| レベル | 用途 | 例 |
|--------|------|-----|
| DEBUG | 開発時のみ有用な詳細 | 変数値、SQL クエリ |
| INFO | 正常な処理の節目 | リクエスト完了、バッチ処理開始/終了 |
| WARN | 異常だが処理は継続可能 | リトライ成功、キャッシュフォールバック |
| ERROR | 処理が失敗し対応が必要 | 決済失敗、DB接続エラー、プロセス継続不可能な致命的障害 |

FATAL を ERROR と分けるかはプロジェクトの方針次第。多くのロギングライブラリやオブザーバビリティツールは ERROR を最高レベルとして扱っており、統一した方がフィルタリングやアラート設計がシンプルになる。

**レベルはシステムの状態を表すものであり、開発者の感覚ではない。** よくある誤用:

| 誤用 | 正しいレベル | 理由 |
|------|------------|------|
| ERROR: ユーザーの入力バリデーション失敗 | WARN | ユーザー操作ミスはシステム障害ではない |
| INFO: 決済失敗 | ERROR | 売上損失の直接影響がある |
| WARN: DB 接続失敗 | ERROR | システム全体の停止リスク |
| ERROR: キャッシュ MISS | WARN | DB フォールバックで処理継続可能 |

外部 API のエラーはリカバリ手段（リトライ、キャッシュ）がある場合は WARN が適切。すべてを ERROR にするとアラート疲れを招く。

**予期する例外 / 予期しない例外の分離**:

| 分類 | 例 | レベル | スタックトレース |
|------|-----|--------|----------------|
| 予期する例外 | バリデーションエラー、業務ルール違反、リトライ可能な外部 API エラー | WARN（場合により INFO） | 不要 |
| 予期しない例外 | NullPointerException、未処理状態、プログラミングミス | ERROR | 必要 |

「対応不要な予期しない例外をいかに減らすか」が AlertStorming（アラート疲れ）回避の本質。例外を出した側でレベル判定するのではなく、**業務ロジックで予期できる失敗は型・戻り値（Result / Either / Option）で表現し、例外は予期しない事象に限定する**。型設計で例外発生自体を予防する（`robust-code.md` の Parse, don't Validate 参照）。

ログレベル戦略はモニタリングポリシーと一緒に開発初期段階で決める。

### 原則 4: 読み手にとって有用な情報を含める

**読者像を意識する**: ログの読者は1種類ではない。読者ごとに必要な情報・粒度は異なる。

| 読者 | 関心事 | 適切な情報 |
|------|-------|-----------|
| エンドユーザー | 自分の操作の結果 | エラーコード、相関 ID（サポート問い合わせ用） |
| 運用者 / SRE | システム健全性、障害の波及範囲 | サービス名、メトリクス、依存先の状態 |
| 開発者 | 不具合の再現と根本原因 | スタックトレース、入力値、内部状態 |
| 監査人 | 誰が何をしたか（Who/When/What/Where） | ユーザー ID、操作対象、IP、タイムスタンプ |

トラブルシューティング目的「だけ」でログを設計しない。監査・業務統計・利用状況分析にも同じログ基盤が使われることを前提にフィールドを設計する。

最低限含めるフィールド:
- **タイムスタンプ**: ISO 8601 形式、タイムゾーン付き
- **ログレベル**
- **識別子**: リクエストID、ユーザーID、セッションID
- **メッセージ**: 何が起きたかの説明（関数名ではなく状況を記述する）

**Wide Event パターン**: 1つのログエントリに十分なコンテキストを詰め込む。複数の薄いエントリより、1つの濃いエントリの方が調査効率が高い。

```json
{
  "level": "info",
  "message": "order_completed",
  "request_id": "req-abc123",
  "user_id": "usr-456",
  "user_tier": "premium",
  "order_id": "ord-789",
  "item_count": 3,
  "total_amount": 15000,
  "payment_method": "credit_card",
  "duration_ms": 234
}
```

ログメッセージの品質基準:
- 何が起きたかが明確に伝わる
- 可能なら対処方法を示唆する
- リファクタリングで壊れない（関数名ではなく状況を記述）

### 原則 5: 機密データを除外する

個人情報（PII）やシークレットをログに含めない。

- マスキングはアプリケーション層で行う（下流のツールに依存しない）
- ログ出力関数をオーバーライドして一律にサニタイズする
- メールアドレス、クレジットカード番号、パスワード、API キーは出力しない
- 本番環境では DEBUG レベルを無効にする（意図せず機密データが含まれやすい）

マスキングの具体パターン:

| データ種別 | マスク形式 | 例 |
|-----------|-----------|-----|
| メールアドレス | 先頭2文字 + `***` + ドメイン | `ad***@example.com` |
| クレジットカード | 末尾4桁のみ | `****-****-****-1234` |
| デバイストークン | 先頭8文字のみ | `a1b2c3d4...` |
| API キー / パスワード | 一切出力しない | — |

## 非機能要求 → 品質特性 → ログ戦術カテゴリ

10カテゴリは並列に存在するのではなく、上位の非機能要求から導出される。新規にログを設計するときは下表で「どの L1 非機能要求のために、どの L2 品質特性を担保し、どの戦術カテゴリに属するか」を辿る。1つのカテゴリが複数の非機能要求に貢献するのが常態（セミラティス）であることに注意。

| 非機能要求（L1 例） | 品質特性（L2） | 戦術（L3） | 該当カテゴリ（L4） |
|------|------|------|------|
| 年間稼働率 99.99% | 可用性 | 異常を検知する | 障害検知 / モニタリング |
| インシデント RTO 30分 | 保守性 | 問題特定を支援する | 障害追跡 / トレーシング |
| SOC2 / PCI DSS 準拠 | セキュリティ・監査性 | 攻撃から回復する／証跡を保全する | 監査 / リモート呼び出し |
| 不正利用検知 | セキュリティ | 攻撃から回復する | 監査 / 障害検知 |
| KPI 日次集計・分析 | 運用性 | 業務状況を可視化する | 業務イベント / 利用統計 |
| 長時間バッチの進捗把握 | 運用性 | 進行状況を可視化する | プログレス / モニタリング |
| 開発時の挙動確認 | テスト容易性 | 内部状態を可視化する | デバッグ |

**読み方**:
- 1行のログが複数行に該当することは普通（例: 監査ログは「セキュリティ」と「コンプライアンス」を同時に満たす）
- 同じ「監査」カテゴリでも、SOC2 準拠目的では保存期間 7年・PII 包含が要件、不正利用検知目的では保存期間 90日・即時アラートが要件、と派生条件が変わる → 後述「同一ログの多目的利用」を参照

## ログの戦術分類（10カテゴリ）

ログの過不足判断は「何のために出すか」を分類することから始まる。各カテゴリで必要項目・実装位置（業務コードに埋めるか、横断的関心事として AOP / ミドルウェアに寄せるか）が異なる。

| カテゴリ | 目的 | 主な必要項目 | 実装位置 |
|---------|------|------------|---------|
| トレーシング | リクエストの処理経路追跡 | trace ID, span ID, メソッド名, 処理時間 | AOP / インターセプター |
| モニタリング | システム健全性の継続観測 | メトリクス源データ（カウント、レイテンシ） | ミドルウェア / メトリクス API |
| リモート呼び出し | 外部システムとの責任分界 | 送信ペイロード、レスポンス、ステータスコード（機密マスク済） | クライアントラッパー |
| 監査（Audit） | セキュリティ・内部統制 | Who（ユーザー ID）/ Where（IP / 端末）/ When / What（操作・対象 ID） | 業務コード明示 |
| 業務イベント | KPI 集計・分析の元データ | ドメインイベント名、ビジネス属性 | 業務コード明示（SemanticLogging） |
| 利用統計 | 機能利用率の集計 | 機能 ID、ユーザー属性 | ミドルウェア |
| 障害追跡 | 個別障害の根本原因調査 | エラー内容、コンテキスト、スタックトレース | フィルタ / 例外ハンドラ |
| 障害検知 | アラート発報のトリガ | 異常イベント名、重大度 | ハンドラ + アラートルール |
| プログレス | 長時間処理の進捗 | 処理対象数、進捗率、推定残時間 | バッチコード明示 |
| デバッグ | 開発時の挙動確認 | 任意の内部状態 | 業務コード（本番では無効） |

**業務コード明示が許されるのは「監査・業務イベント・プログレス」の3つに限定するのが基本**。それ以外は AOP・インターセプター・ミドルウェア・例外ハンドラに寄せる。業務コードに「業務ロジック + ログ呼び出し」が混在すると、粒度が担当者ごとにバラつき、横断分析が困難になる。

## 同一ログの多目的利用（セミラティス構造の実務的帰結）

セミラティス構造では1本のログが複数の非機能要求を同時に満たしうる。これはコスト効率的だが、目的ごとに要求される **保存期間・PII 取扱・サンプリング・アラート粒度** が衝突するため、設計時に明示的に解決する必要がある。

### 目的軸ごとに変わる設計パラメータ

| パラメータ | 監査（コンプライアンス） | 障害追跡（保守性） | 業務分析（KPI） | 障害検知（可用性） |
|----------|----------------------|-----------------|---------------|-----------------|
| 保存期間 | 7年（規制依存） | 30〜90日 | 1〜3年（集計後は集計値のみ） | 7〜30日 |
| PII 取扱 | **含める必要あり**（誰が何をしたか） | マスク（個人特定不要） | 匿名化・集計 | マスクまたは除外 |
| サンプリング | 不可（全件） | 可（エラー時は全件、正常はサンプル） | 不可（集計の正確性） | 不可（検知漏れ防止） |
| 改ざん耐性 | 必須（WORM・ハッシュ連鎖） | 不要 | 不要 | 不要 |
| 即時性 | 不要 | 数分〜数時間 | バッチ（日次） | **秒〜分**（アラート） |
| アクセス権限 | 監査人・限定 | 開発者・SRE | 分析チーム | SRE・全員可 |

### 衝突を解決する設計選択肢

1. **ログを目的別に分離する**: 監査ログと障害追跡ログを別ストリームに分け、保存先・保存期間・アクセス権限を独立させる。最も安全だが運用コストは高い
2. **同一ログに目的タグを付与し、下流で振り分ける**: `purpose: ["audit", "troubleshooting"]` のようにフィールドで多目的を表明し、収集基盤側でルーティング。共通フィールドの粒度合意が前提
   ```jsonc
   // 一次ログ出力（目的タグで多目的を表明）
   { "event": "order_placed", "purpose": ["audit", "business_event"], "actor_id": "...", "order_id": "...", ... }
   // 収集側ルーティング: purpose に "audit" を含むレコードはアーカイブストレージへ、
   //                   "business_event" を含むレコードは分析用ストリームへ複製
   ```
3. **マテリアライズドビュー / 派生ストリーム**: 一次ログから目的別に派生ログを生成（PII 除去版を分析チームへ、フル版を監査用ストレージへ）。PII 境界を物理的に切れる

**判断基準**: PII を含むかつ複数の読み手がいる場合は (1) または (3) を選ぶ。読み手が同質（運用者のみ等）なら (2) で十分。`原則 5: 機密データを除外する` と整合させること。

### よくある失敗

- **「監査にも使えそうだから保存期間を7年にする」**: 障害追跡ログにも7年適用 → ストレージコスト爆発。目的タグで切り分けるべき
- **「分析にも使うから PII を残す」**: 規制違反リスク。分析用は派生時に匿名化する
- **「障害検知ログを業務分析にも流用」**: サンプリング有無の要求が衝突。検知用はサンプルしないが、分析用はサンプルできる場合がある → 別ストリームに

## 実装パターン

### ロガーファサード経由で出力する

`println` / `printf` / 直接 `fs.write` で自前ログを書かない。ベンダーロックインを避けるため、ファサード（slf4j、pino、zap、log/slog 等）経由で出力し、バックエンド実装は設定で差し替え可能にする。

### 自前ローテーションを書かない

ファイルローテーション・圧縮・転送はランタイム（logrotate、Fluent Bit、コンテナの stdout 収集）に任せる。アプリケーションはストリーム（標準出力 / ファイル）に書き出すだけにする。これは 12-factor App の "Logs as event streams" 原則そのもの。

### MDC でコンテキストを伝搬する

横断的コンテキスト（リクエスト ID、ユーザー ID、テナント ID）は実行コンテキストに格納し、ログ出力時に自動付与する。

| 言語 | 実装 |
|------|------|
| Java | MDC（slf4j） |
| Node.js | AsyncLocalStorage |
| Python | `contextvars` |
| Go | `context.Context`（slog の Handler でフィールド注入） |
| Rust | `tracing` の Span |

ハンドラ／ミドルウェアの入口でセットすれば、配下のすべてのログ出力で透過的に参照される。関数引数で引き回さない。

### 横断的ロギングは AOP / ミドルウェアに寄せる

入退場ログ・例外ログ・トレーシングは業務コードに書かず、以下に集約する。

- HTTP: ミドルウェア / インターセプター
- gRPC: ServerInterceptor
- バッチ: テンプレートメソッド / Decorator
- メソッド単位: AOP（Java AspectJ、Python decorator 等）

業務コードに残るログは「業務イベント」「監査」「プログレス」のみ、というのが目安。

### フォーマット・キー名を統一する

チーム内でフィールド名（`request_id` か `requestId` か等）と必須キーを規約化する。Logger ラッパーで強制するか、lint で検査する。粒度・命名の不統一は分析時の致命傷になる。

## オブザーバビリティとの統合

ログ単体ではなく、メトリクス・トレースと組み合わせて初めてシステムの全体像が見える。

### テレメトリの3本柱

| シグナル | 役割 | 粒度 |
|----------|------|------|
| **メトリクス** | システム全体の健全性・傾向 | 集約値（CPU、エラー率、キュー深度） |
| **トレース** | リクエストのサービス横断的な処理経路 | リクエスト単位（trace ID + span ID） |
| **ログ** | 特定イベントの詳細コンテキスト | イベント単位 |

### 相関分析のフロー

メトリクスで異常を検知 → トレースで該当リクエストの処理経路を特定 → ログで根本原因を調査。

この連携を機能させるために:
- ログに **trace ID / span ID** を含める（OTel のコンテキスト伝搬を利用）
- 正常な処理はスパンのアトリビュートとして表現し、ログは異常時（WARN 以上）に出力する
- 長時間処理では処理の節目で INFO ログを出す

### ログベースのモニタリング

構造化ログからメトリクスを導出するパターン:

1. ログを集約ストレージに収集
2. フィールド条件でフィルタリング・カウント → 数値メトリクスに変換
3. 閾値を超えたらアラート発報

**アラートアクションの5段階分類**:

| アクション | 意味 | 通知チャンネル例 |
|-----------|------|----------------|
| ALERT_IMMEDIATE | 1回発生で即時対応 | PagerDuty + Slack |
| ALERT_THRESHOLD | 時間窓内で閾値超過時に通知 | Slack + Email |
| ALERT_TREND | 長期的な悪化傾向で通知 | Email + Dashboard |
| DASHBOARD_ONLY | ダッシュボード表示のみ | Dashboard |
| NO_ACTION | 監査・デバッグ用の保存のみ | — |

アラート設計の注意点:
- 単発のエラーではなく、時間窓内での発生回数で判定する（リトライ機構がある場合）
- メッセージ文字列ではなくフィールド値でフィルタする（コード変更でアラートが壊れない）
- ログカテゴリごとにアクションレベルを決めておく（例: DB接続失敗 → IMMEDIATE、通常クエリログ → NO_ACTION）

## アンチパターン

| アンチパターン | 何が問題か | 正しい対処 |
|--------------|----------|----------|
| **役に立たないログの過剰出力** | ストレージ・スループット・読み手の認知負荷を消費するだけで何も得られない | 「インシデント調査で困るか」で削除候補を判定（原則1） |
| **粒度の不統一** | 担当者ごとにキー名・出力箇所がバラバラで横断分析できない | フィールド名を定数 / enum で集中管理。Logger ラッパーで強制 |
| **AlertStorming（アラート疲れ）** | 偽陽性アラートが大量発生し、真の障害が埋もれる | 予期する例外は WARN 以下、ERROR は対応必須のみ。アラートは時間窓 + フィールド条件で発火 |
| **業務コードへのログ散在** | 業務ロジックが「ログ呼び出し」で汚染され、横断的関心事の変更コストが高くなる | AOP / ミドルウェアへ寄せる。業務コードに残すのは監査・業務イベント・プログレスのみ |
| **メッセージ文字列での状態判定** | メッセージ変更でアラート・ダッシュボードが壊れる | 状態は構造化フィールド（イベント名 + 属性）で表現 |
| **自前ファイル I/O・自前ローテーション** | 信頼性が低く、運用責務がアプリに漏れる | ファサード経由でストリームに出力し、外部ローテータに委譲 |
| **ベンダー固有ロガーの直接利用** | 移行コスト・テスト容易性が下がる | ファサード（slf4j、pino、zap、log/slog 等）経由で抽象化 |
| **本番 DEBUG 有効化** | 機密データ漏洩リスク・パフォーマンス劣化 | 本番は INFO 以上。一時的に上げる場合はサンプリング + 期限付き |
| **「予期しない例外」の放置** | 障害追跡ログが ERROR で溢れアラート疲れに繋がる | 型設計・契約による設計（Result / Either / Option）で例外発生自体を予防 |

## チェックリスト

ログ設計のレビュー時に確認する。

### 設計

- [ ] 各ログが L1 非機能要求まで遡れる（なぜ存在するかが言語化されている）
- [ ] 同一ログが複数目的で使われる場合、保存期間・PII 取扱・サンプリングの衝突が解決されている
- [ ] ログ出力箇所が明確で重複がない
- [ ] Canonical Log Line パターンが適用されている（またはその判断理由がある）
- [ ] 各ログが10カテゴリ（戦術分類）のいずれかに分類されている
- [ ] 業務コード内のログは監査・業務イベント・プログレスに限定されている
- [ ] ログレベルがモニタリングポリシーと整合している
- [ ] 予期する例外（WARN）と予期しない例外（ERROR）が分離されている
- [ ] 構造化ログを使い、データはフィールドに分離されている

### コンテンツ

- [ ] タイムスタンプ・ログレベル・識別子（request ID 等）が含まれている
- [ ] 想定読者（運用者・開発者・監査人）に対して必要な情報が揃っている
- [ ] メッセージは状況を記述し、関数名に依存していない
- [ ] 機密データ（PII、シークレット）が含まれていない
- [ ] フィールド名が定数 / enum で管理されている

### 実装

- [ ] ロガーファサード経由で出力している（自前 I/O していない）
- [ ] MDC / AsyncLocalStorage 等でコンテキストが自動付与されている
- [ ] 横断的ロギング（入退場・例外・トレース）が AOP / ミドルウェアに寄せられている
- [ ] フィールド名・必須キーの規約があり、Logger ラッパーまたは lint で強制されている
- [ ] ローテーション・転送はランタイム（logrotate / Fluent Bit / stdout 収集）に委譲している

### オブザーバビリティ

- [ ] trace ID / span ID がログに含まれている（分散システムの場合）
- [ ] 正常処理はスパン、異常はログという使い分けが考慮されている
- [ ] ログベースのアラートがフィールド値で定義されている
- [ ] アラートが時間窓 + フィールド条件で発火し、AlertStorming を防いでいる
